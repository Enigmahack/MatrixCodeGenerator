#1 — Pre-allocate Occupancy Arrays (WebGLRenderer.js ~L1549, L1694)

  Bottleneck: _renderQuantizedLineGfx() creates new Uint8Array(gw * gh * 4) every frame while a Quantized effect is active —
   ~34KB allocated and GC'd at 60fps.

  Fix: Pre-allocate once when the grid is initialized and fill(0) to reset instead. This eliminates GC pressure, which
  causes the worst frame spikes on low-end hardware.

  ---
  #2 — Full Instance Buffer Rebuild Every Frame (WebGLRenderer.js ~L1968–2137)

  Bottleneck: Every frame, the renderer loops over all cols × rows cells (~2,160+) to rebuild the instance buffer from
  scratch, then uploads the entire ~86KB to the GPU via bufferSubData — even if only 5% of cells changed.

  Fix: Implement a dirty-cell bitmask. Track which cells changed in the simulation tick and only rewrite + upload those.
  Could reduce GPU upload cost by 70–90% on typical frames.

  ---
  #3 — Per-Cell Color Blending with Math.floor + Bitwise Ops (SimulationSystem.js ~L564–588)

  Bottleneck: Every active cell's color fade does 8 bitwise shifts/masks + 3 Math.floor() calls per interpolation step. On
  lower-end ARM CPUs (budget laptops, older machines), this is not vectorized and hits hard in the update loop.

  Fix: Replace with pre-packed integer math using bitwise rounding (>> 0 or | 0 instead of Math.floor), or pre-compute a
  color blend LUT indexed by (fromColor, toColor, step) for the limited discrete values actually used. Also consider
  batching inactive cells out of the update loop entirely.

  ---
  Priority order for lower-end impact: #2 > #1 > #3. The full buffer rebuild is the most consistently expensive, occupancy
  allocations cause the worst intermittent spikes, and color math is the most CPU-bound on ARM/Atom-class hardware.
